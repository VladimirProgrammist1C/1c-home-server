## 1. ОБЩИЕ СВЕДЕНИЯ <a id="1-общие-сведения"></a>

### 1.1. Назначение документа
Настоящая техническая спецификация определяет требования к разработке, структуре метаданных, алгоритмам и интерфейсам расширения `ВЧ_ИнтеграцияСVoceChat` для платформы 1С:Предприятие 8.3. Документ служит руководством для разработчиков и тестировщиков.

### 1.2. Область применения
- **Конфигурация-основание:** 1С:Зарплата и управление персоналом 3.1 (редакция 3.1.34+)
- **Платформа:** 1С:Предприятие 8.3.20+ (рекомендуется 8.5.11+)
- **Режим работы:** Клиент-серверный / Файловый
- **Тип поставки:** Расширение конфигурации (`.cfe`)

### 1.3. Соответствие бизнес-требованиям
Спецификация реализует архитектурное решение, утверждённое в `02_ARCHITECTURAL_DECISION.md`:
- ✅ Автоматическая отправка уведомлений при первом проведении «Прием на работу»
- ✅ Управление интеграцией через глобальную константу (без модификации типовых форм)
- ✅ Асинхронная очередь с механизмом повторных попыток
- ✅ Нулевое влияние на метаданные типовой конфигурации

## 2. АРХИТЕКТУРА РЕШЕНИЯ <a id="2-архитектура-решения"></a>

### 2.1. Компоненты расширения
Расширение реализует паттерн **Producer-Consumer**:
- **Producer:** Подписка на событие `ПередЗаписью` документа «Прием на работу». Формирует задачу и помещает её в регистр очереди.
- **Consumer:** Регламентное задание. Вычитывает задачи из очереди, выполняет HTTP-запрос к VoceChat Bot API, обновляет статус.

### 2.2. Схема взаимодействия
```
[Документ «Прием на работу»]
          │
          ▼ (ПередЗаписью)
[ВЧ_ОбработчикиСобытий] → Проверка флага ВЧ_ИспользоватьИнтеграцию
          │
          ▼
[Регистр сведений ВЧ_ОчередьУведомлений] ← Запись задачи (Статус: Ожидает)
          │
          ▼ (Каждые 60 сек)
[Регламентное задание ВЧ_ОбработчикОчередиУведомлений]
          │
          ▼
[ВЧ_ИнтеграцияСервер.ОтправитьВVoceChat()] → HTTP POST → VoceChat API
          │
          ├──▶ Успех (200) → Статус: Отправлено
          └──▶ Ошибка → Попытки++, Статус: Ошибка (после лимита)
          │
          ▼
[Журнал регистрации] ← Фиксация результата
```

### 2.3. Принципы реализации
- **Изоляция:** Вся логика инкапсулирована в общих модулях расширения.
- **Асинхронность:** Проведение документа не блокируется ожиданием ответа от мессенджера.
- **Надёжность:** Использование `ХранилищеЗначения` для текста сообщений исключает потерю данных при обрезке строк.
- **Совместимость:** Отсутствие изменений в типовых объектах гарантирует безопасное обновление конфигурации-основания.

## 3. СТРУКТУРА ОБЪЕКТОВ МЕТАДАННЫХ <a id="3-структура-объектов-метаданных"></a>

### 3.1. Общие модули
| Имя | Глобальный | Сервер | Клиент | Назначение |
|-----|------------|--------|--------|------------|
| `ВЧ_ИнтеграцияСервер` | Нет | ✅ | ❌ | HTTP-взаимодействие с VoceChat API, работа с хранилищем |
| `ВЧ_ОбработчикиСобытий` | Нет | ✅ | ❌ | Подписка на события, управление очередью, регламентное задание |

### 3.2. Константы
| Имя | Тип | Описание |
|-----|-----|----------|
| `ВЧ_ИспользоватьИнтеграцию` | Булево | Глобальный переключатель активности интеграции |
| `ВЧ_URLСервера` | Строка | Адрес сервера VoceChat (без протокола) |
| `ВЧ_ПортСервера` | Число | Порт сервера (по умолчанию 3001) |
| `ВЧ_BotAPIKey` | Строка | Ключ авторизации бота (маскируется в форме) |
| `ВЧ_IDКанала` | Число | Идентификатор целевого канала для отправки |
| `ВЧ_ТаймаутHTTP` | Число | Таймаут соединения в секундах (по умолчанию 30) |
| `ВЧ_МаксимальноеКоличествоПопыток` | Число | Лимит повторных отправок при ошибках (по умолчанию 3) |
| `ВЧ_ОтветственныйКадровик` | СправочникСсылка.Пользователи | Ответственный по умолчанию для уведомлений |

### 3.3. Регистры сведений
| Имя | Тип | Периодичность | Измерения | Ресурсы |
|-----|-----|---------------|-----------|---------|
| `ВЧ_ОчередьУведомлений` | Независимый | **В пределах секунды** | `ИДСообщения` (Строка, 50)<br>`IDКанала` (Число, 10)<br>`ДокументОснование` (ДокументСсылка.ПриемНаРаботу) | `Статус` (ПеречислениеСсылка.ВЧ_СтатусыУведомлений)<br>`ТекстСообщения` (ХранилищеЗначения)<br>`ПопыткиОтправки` (Число, 2)<br>`ТекстОшибки` (Строка, 500)<br>`ТипСобытия` (Строка, 50)<br>`Параметры` (ХранилищеЗначения)<br>`ДатаСоздания` (Дата)<br>`ДатаПоследнейПопытки` (Дата)<br>`ОтветственныйКадровик` (СправочникСсылка.Пользователи) |

**Примечание:** Наличие измерений `IDКанала` и `ДокументОснование` позволяет эффективно строить отборы и отчеты по истории уведомлений без полного сканирования регистра.

### 3.4. Перечисления
| Имя | Значения | Назначение |
|-----|----------|------------|
| `ВЧ_СтатусыУведомлений` | `Ожидает`, `Отправлено`, `Ошибка` | Управление жизненным циклом задачи в очереди |

### 3.5. Подписки на события
| Имя | Событие | Объект | Обработчик |
|-----|---------|--------|------------|
| `ВЧ_ПриПриемеНаРаботу` | ПередЗаписью | `Документ.ПриемНаРаботу` | `ВЧ_ОбработчикиСобытий.ПриПриемеНаРаботуПередЗаписью` |

### 3.6. Регламентные задания
| Имя | Интервал | Метод | Назначение |
|-----|----------|-------|------------|
| `ВЧ_ОбработчикОчередиУведомлений` | **30 секунд** | `ВЧ_ОбработчикиСобытий.ОбработатьОчередь()` | Вычитка и отправка задач со статусом «Ожидает» |

### 3.7. Интерфейсные объекты
| Объект | Тип | Назначение |
|--------|-----|------------|
| `ВЧ_НастройкиИнтеграции` | Общая форма | Просмотр и редактирование констант, управление флагом активности |
| `ВЧ_ТестированиеОтправки` | Общая команда | Открытие обработки тестовой отправки |
| `ОтправкаСообщенийVoceChat` | Встроенная обработка | Валидация настроек, ручная отправка тестового сообщения, диагностика соединения |

### 3.8. Роли
| Роль | Права | Назначение |
|------|-------|------------|
| `ВЧ_ОсновнаяРоль` | Чтение констант, запись в регистр очереди | Базовые пользователи (кадровики) |

| `ВЧ_ЧтениеИнтеграции` | Только чтение | Аудит, мониторинг статусов |

## 4. АЛГОРИТМЫ ОБРАБОТКИ <a id="4-алгоритмы-обработки"></a>

### 4.1. Обработчик события «Прием на работу»
**Метод:** `ВЧ_ОбработчикиСобытий.ПриПриемеНаРаботуПередЗаписью(Источник, Отказ)`

1. Проверить `ВЧ_ИспользоватьИнтеграцию`. Если `Ложь` → завершить.
2. Проверить условия триггера:
   - `Источник.ЭтоНовый()` И `Не Источник.Проведен`
   - Заполнены обязательные поля: `Сотрудник`, `ПодразделениеОрганизации`, `Должность`
3. Сформировать текст уведомления по шаблону (см. Приложение А).
4. Создать запись в `ВЧ_ОчередьУведомлений`:
   - `ИДСообщения` = Уникальный идентификатор (UUID или код документа + таймстамп)
   - `Статус` = `Ожидает`
   - `ТекстСообщения` = Поместить в `ХранилищеЗначения`
   - `ПопыткиОтправки` = 0
   - `ДатаСоздания` = Текущая дата
5. Зафиксировать в журнале регистрации: *«Задача на отправку уведомления создана»*.

### 4.2. Обработка очереди (регламентное задание)
**Метод:** `ВЧ_ОбработчикиСобытий.ОбработатьОчередь()`

1. Выбрать записи из `ВЧ_ОчередьУведомлений` с условием:
   - `Статус = Ожидает`
   - `ПопыткиОтправки < ВЧ_МаксимальноеКоличествоПопыток`
2. Для каждой записи:
   - Извлечь текст из `ХранилищеЗначения`
   - Вызвать `ВЧ_ИнтеграцияСервер.ОтправитьСообщение(Текст, URL, Порт, Ключ, Канал, Таймаут)`
   - **При успехе (HTTP 200):**
     - `Статус` = `Отправлено`
     - Записать в журнал: *«Уведомление успешно отправлено»*.
   - **При ошибке (HTTP != 200 или исключение):**
     - `ПопыткиОтправки`++
     - `ТекстОшибки` = Код состояния + текст исключения
     - Если `ПопыткиОтправки >= Лимит` → `Статус` = `Ошибка`
     - Записать в журнал: *«Ошибка отправки. Попытка N из M»*.
3. Записать изменения в регистр одной транзакцией.

### 4.3. HTTP-взаимодействие с VoceChat
**Метод:** `ВЧ_ИнтеграцияСервер.ОтправитьСообщение(...)`

1. Сформировать URL: `http://{URL}:{Порт}/api/bot/send_to_group/{Канал}`
2. Создать `HTTPСоединение` с указанным таймаутом.
3. Сформировать `HTTPЗапрос`:
   - Метод: `POST`
   - Заголовки: `x-api-key: {Ключ}`, `Content-Type: text/markdown; charset=utf-8`
   - Тело: Текст сообщения (UTF-8)
4. Выполнить запрос, получить `HTTPОтвет`.
5. Вернуть структуру: `{Успех: Булево, КодСостояния: Число, Текст: Строка}`.

## 5. ОБРАБОТКА ОШИБОК И ЛОГИРОВАНИЕ <a id="5-обработка-ошибок-и-логирование"></a>

### 5.1. Стратегия повторных попыток
- Максимальное количество попыток задаётся константой `ВЧ_МаксимальноеКоличествоПопыток`.
- Интервал между попытками определяется частотой регламентного задания (30 секунд).
- Задачи со статусом `Ошибка` не обрабатываются автоматически. Требуется ручной анализ или очистка очереди.

### 5.2. Журнал регистрации
Все события фиксируются в журнале с именем события `ВЧ_Интеграция`:

| Уровень | Сценарий | Пример текста |
|---------|----------|---------------|
| `Информация` | Успешная отправка | `Уведомление отправлено в канал {ID}. Сотрудник: {ФИО}` |
| `Предупреждение` | Повторная попытка | `Ошибка HTTP 503. Попытка 2 из 3. Задача {ИД}` |
| `Ошибка` | Исчерпан лимит попыток | `Отправка прервана. Лимит попыток исчерпан. Текст: {Ошибка}` |
| `Ошибка` | Некорректные настройки | `Неверный BotAPIKey (HTTP 401). Проверьте константы.` |

### 5.3. Восстановление после сбоев
- При перезапуске сервера 1С очередь сохраняется в базе данных.
- После восстановления связи с VoceChat регламентное задание автоматически продолжит обработку задач со статусом `Ожидает`.
- Администратор может вручную изменить статус записей в регистре через универсальный отчет или обработку.

## 6. ТРЕБОВАНИЯ К БЕЗОПАСНОСТИ <a id="6-требования-к-безопасности"></a>

### 6.1. Разграничение прав
- Доступ к константам настройки осуществляется только через роль `ВЧ_АдминистраторИнтеграции`.
- В форме `ВЧ_НастройкиИнтеграции` поле `ВЧ_BotAPIKey` имеет тип отображения `Пароль` (маскировка символов).
- Запись в регистр очереди доступна пользователям с ролью `ВЧ_ОсновнаяРоль`.

### 6.2. Защита данных
- Ключи API не выводятся в лог в открытом виде.
- Текст сообщений хранится в `ХранилищеЗначения`, что предотвращает усечение длинных строк и повышает целостность данных.
- Не передаются персональные данные сверх необходимых для идентификации сотрудника (ФИО, должность, телефон).

### 6.3. Сетевая безопасность
- Требуется разрешение брандмауэра на исходящие соединения с сервера 1С на порт VoceChat.
- Рекомендуется использовать HTTPS, если сервер VoceChat настроен с SSL-сертификатом.
- Валидация SSL-сертификата выполняется средствами платформы 1С.

## 7. ТЕСТИРОВАНИЕ И ПРИЁМКА <a id="7-тестирование-и-приёмка"></a>

### 7.1. Сценарии тестирования
| ID | Сценарий | Предусловия | Ожидаемый результат |
|----|----------|-------------|---------------------|
| TC-01 | Первое проведение «Прием на работу» | Интеграция включена, настройки заполнены | Запись в регистре со статусом `Ожидает` → через ≤30 сек статус `Отправлено`, сообщение в VoceChat |
| TC-02 | Повторное проведение того же документа | Документ уже проведен ранее | Новая запись в регистре НЕ создается |
| TC-03 | Интеграция выключена | `ВЧ_ИспользоватьИнтеграцию = Ложь` | Проведение документа завершается успешно, запись в регистре НЕ создается |
| TC-04 | Неверный BotAPIKey | В константе указан невалидный ключ | Статус записи меняется на `Ошибка` после лимита попыток, в журнале запись с кодом 401 |
| TC-05 | Сервер VoceChat недоступен | Сетевой доступ закрыт или сервис остановлен | Статус записи меняется на `Ошибка` после лимита попыток, задача не теряется |
| TC-06 | Тестовая отправка | Запущена обработка `ОтправкаСообщенийVoceChat` | При успешном соединении выводится сообщение «Тест пройден», в канал приходит сообщение |
| TC-07 | Изменение настроек | Открыта форма `ВЧ_НастройкиИнтеграции` | Значения сохраняются в константы, интерфейс синхронизирован, чувствительные поля маскируются |

### 7.2. Критерии приёмки
- [ ] Все сценарии TC-01 – TC-07 выполнены успешно.
- [ ] Журнал регистрации содержит корректные записи с уровнем `Информация`/`Ошибка`.
- [ ] Регламентное задание выполняется без ошибок в фоне.
- [ ] Обновление типовой конфигурации 1С:ЗУП не приводит к конфликтам метаданных.
- [ ] Документация актуальна и соответствует коду.

## 8. РАЗВЁРТЫВАНИЕ И ОБНОВЛЕНИЕ <a id="8-развёртывание-и-обновление"></a>

### 8.1. Установка
1. Скопировать файл `ВЧ_ИнтеграцияСVoceChat_v1.0.cfe` на сервер 1С.
2. В Конфигураторе: **Конфигурация → Расширения конфигурации → Добавить**.
3. Выбрать файл `.cfe`, нажать **F7** для обновления БД.

4. В режиме Предприятие открыть форму `ВЧ_НастройкиИнтеграции`:
   - Активировать переключатель `ВЧ_ИспользоватьИнтеграцию`.
   - Заполнить параметры подключения (поля станут доступны после активации).
   - Проверить связь кнопкой «Тестирование».
5. Назначить пользователям соответствующие роли.

### 8.2. Обновление
- При выпуске новых версий `.cfe` достаточно заменить файл расширения в Конфигураторе и обновить конфигурацию БД.
- Константы и регистры сохраняются между версиями.
- Перед обновлением рекомендуется сделать резервную копию базы.

### 8.3. Совместимость
- Расширение не модифицирует типовые объекты, что гарантирует безопасное обновление конфигурации-основания от вендора.
- При изменении структуры типовых объектов, на которые есть ссылки (например, переименование реквизитов), потребуется адаптация кода расширения.
